# Architecture

> A USD price flows through one detector and one renderer to produce a
> "≈ N nets 🦟" pill. Same logic runs in three shells: the web
> converter, the bookmarklet IIFE, and the Chrome extension content
> script.

## Workspace map

```
packages/
  charities/        Validated reference data + math + price/currency parsing
                    (Zod schemas, charity list, parsePriceString, toUsd)
apps/
  bookmarklet/      Detector + renderer + IIFE build pipeline
                    Bundles itself into apps/web/src/generated/bookmarklet.ts
  web/              Next.js converter UI + drag-link surface
  extension/        Chrome MV3 content script + manifest + build
                    Imports detector/renderer from bookmarklet workspace
```

## Data flow

```
Page DOM ──► Detector ──► [DetectedPrice...] ──► convertPrice ──► formatUnits
                            (priceUsd, anchorEl)                      │
                                                                      ▼
                                                                 renderPill
                                                                  (target,
                                                                   label)
```

## The two contracts

Both live in `apps/bookmarklet/src/types.ts`.

```ts
interface DetectedPrice {
  readonly priceUsd: number;   // detectors normalise; only USD for v1
  readonly anchorEl: Element;  // where the pill is placed
}

interface Detector {
  readonly id: string;
  matches(url: URL): boolean;
  detect(root: ParentNode): readonly DetectedPrice[];
}
```

Anything that picks prices off a page implements `Detector`. Anything
that renders the result uses `renderPill(target, label)` from
`apps/bookmarklet/src/render.ts`.

## Where to add things

| Task | Workspace | File |
| --- | --- | --- |
| New charity | charities | `src/index.ts` (the `charities` array) |
| New currency | charities | `src/parsePrice.ts` (CURRENCY_TOKENS) + `src/fx.ts` (fxRates) |
| New site detector | bookmarklet | `src/detectors/<site>.ts` + register in `runtime.ts` |
| Different badge style | bookmarklet | `src/render.ts` (PILL_STYLE constant) |
| UI changes | web | `src/app/page.tsx` or `src/components/` |
| Extension-only logic | extension | `src/content.ts` |

## Why three shells?

| Shell | Reach | Tradeoff |
| --- | --- | --- |
| Web converter | Anyone with a URL | Manual paste. No in-context annotation. |
| Bookmarklet | Anyone, any site | Single inline IIFE. Capped by `javascript:` URL size (~78 KB today, dominated by Zod runtime). |
| Extension | Chrome only | Best UX (auto-runs, MutationObserver), needs install step. |

The shells share *all* domain logic. Differences are only in how the
runtime is bundled, loaded, and triggered.

## Generated files

- `apps/web/src/generated/bookmarklet.ts` — the minified IIFE as a
  string export. Regenerated by `bun run build:bookmarklet`. Gitignored.
- `apps/extension/dist/` — `content.js`, `manifest.json`, `options.html`.
  Produced by `bun --filter '@price-to-impact/extension' run build`.
  Gitignored. Load via `chrome://extensions` → "Load unpacked".

Root scripts (`dev`, `build`, `typecheck`) regenerate the bookmarklet
bundle through `prebuild` / `predev` / `pretypecheck` hooks, so the
gitignored output is always fresh when consumers read it.
